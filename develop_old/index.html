    项目定位¶

    本地化、隐私导向的 AV 领域元数据管理与情报分析系统。目标是收藏与找出最有艺术性的作品。说实话大部分作品都是垃圾。由于独特的性癖，这个定制化程度非常的高。

    • Insight (洞察) ：不仅仅是存数据，而是发现数据背后的联系（如：挖掘“共演关系”、“新人出道轨迹”）。
    • Immersive (沉浸) ：从“填表单”转变为“刷流媒体”般的体验。
    • Zero-Friction (零摩擦) ：极致的自动化采集，让用户感知不到录入过程。

    核心功能矩阵： 1. 数据采集与清洗 (Intelligence Gathering) ： - 多源爬虫 ：集成了 JavBus, JavDB, Minnano, Fanza 等多个数据源的爬虫 ( core/crawler )。 - 浏览器协同 ：通过 Firefox 插件 ( extensions/firefox_capture ) 与本地服务器 ( server/app.py ) 联动，实现了“浏览即采集”的高效工作流。交互能力，在浏览器中就知道这个片子本地是否收藏过 - ID 解析与清洗 ：具备处理复杂番号、去重的能力。 2. 档案管理 (Asset Management) ： - 结构化存储 ：公有库管理“女优 (Actress)”和“作品 (Work)”，“男优 (Actor)”，“标签”等等核心实体，及其关联关系。私有库关系喜欢的女优和作品，包括个人数据例如撸管记录。 - 开放导出，所有的数据都可自由导出为CSV格式。 3. 可视化与洞察 (Visualization & Insight) ： - 关系图谱 ： ForceDirectedView (力导向图) 是产品的核心差异化亮点，用于展示人物与作品、人物与人物之间的关联网络。 - 多维视图 ：包含时间轴 ( TimeLinePage )、画廊 ( GalleryPage ) 和列表视图。

    量化自我 (Quantified Self) —— 贤者模式报告 ： - 改进 ：不要只记录“看了什么”，要分析“口味偏好变化”。生成年度/月度报告： - "你这个月偏爱 MILF 类型，比上个月增加了 30%。" - "你在周五晚上的活跃度最高。" - 价值 ：利用数据可视化（ForceView 之外的图表），让用户通过数据重新认识自己的性癖好。这能带来极强的情感连接。 隐私向，一键伪装功能，按下快捷键，界面变成Excel界面

    独特UI风格，为CJK用户设计，展示页独特的竖排文本，体现CJK文化。极致的文化契合度 ：日文和中文（尤其是繁体）在竖排时能传达出强烈的“文学感”、“高级感”和“传统美学”。对于展示日系作品（AV 元数据），这能带来原汁原味的沉浸体验，仿佛在阅读日式海报或杂志。

    i18n,目标支持多语言，包括简中、繁中、日文、韩文。

    痛点 ： - 工具属性过重 ：目前的 UI 逻辑偏向“数据库增删改查”工具，缺乏“消费内容”的沉浸感。 - 交互割裂 ：桌面端和浏览器端虽然有连接，但体验上是“采集 -> 传输 -> 刷新查看”，不够流畅。 - 录入繁琐 ： 用户更期待“一键完成”。

    引入“Inbox（收件箱）”概念 ： - 浏览器插件采集的数据不再直接入库，而是进入“待处理区”。 - 桌面端显示一个类似 Trello 的看板或流式列表。 - 智能预处理 ：系统自动后台跑完爬虫、人脸识别、NSFW 检测。用户只需要做最后一步：“确认归档”。

    1. 架构层面 (Architecture)¶

    目前项目采用的是 Monolithic (单体) 架构，混合了 MVC 和 MVVM 模式。

    • 问题 1：MainWindow 职责过重 (God Class)

    • 现状 ： MainWindow 类（ ui/main_window.py ）承担了过多的职责，包括路由跳转、信号连接、服务初始化（ TaskManager , StatusManager ）、全局快捷键注册等。

      • 依赖注入 (Dependency Injection) ：目前服务是在 MainWindow 内部直接实例化的。建议引入一个简单的 ServiceLocator 或 Context 类，负责管理 StatusManager 、 TaskManager 、 DatabaseManager 等单例服务， MainWindow 只需向容器请求即可。
      • 问题 2：数据访问层 (DAL) 耦合

    • 现状 ： core/database/query.py 中充斥着大量的原始 SQL 拼接函数，且直接返回 dict 或 tuple 。虽然有 repositories/WorkRepository.py 文件，但似乎为空或未被充分利用。

    • 优化建议 ：

      • 完善 Repository 模式 ：将 query.py 中的函数迁移到对应的 Repository 类中（如 WorkRepository , ActressRepository ）。
      • 引入实体模型 (Domain Models) ：不要在 UI 层直接操作数据库返回的字典。建立明确的 Python 类（如 class Work , class Actress ）或使用 dataclasses / pydantic ，确保类型安全和代码提示友好。

    3. 代码质量与规范 (Code Quality)¶

    • 规范化配置管理 ：
    • 代码中存在硬编码的样式和颜色（如 #F7E6B0 , #282c34 ）。建议建立 ui/theme.py 或 resources/styles/theme.json 统一管理颜色和尺寸常量，方便后续支持“暗黑模式”或换肤。
    • 异常处理 ：
    • core/database/query.py 中的 SQL 执行部分虽然使用了 with 语句管理连接，但部分查询缺少细致的 try-except 捕获，一旦 SQL 报错可能会导致程序崩溃或状态异常。
    • 类型提示 (Type Hinting) ：

    • 部分新代码使用了 Type Hints，建议推广到全项目，特别是核心逻辑部分，有助于减少运行时错误。

    • P1 (中优先级) ： 重构数据层 。停止在 query.py 中写裸 SQL，迁移到 Repository 模式，并返回对象而非字典。

    • P2 (低优先级) ： UI 样式统一 。提取颜色和样式常量，规范化 QSS 管理。

    你这个要求的关键点其实不是“性能”本身，而是：
    加/减点和边之后，原有布局基本不动，只在局部微调 。
    这一点决定了我们必须做“增量维护”，而且还要让 SimulationState 里的数据结构支持长期演化，而不是每次重随机化。
