# darkeye_ui 组件库修改意见

更新时间：2026-03-07

## 1. 现状结论（基于当前代码扫描）

- `darkeye_ui/components` 下共有 40 个组件文件。
- 15 个组件文件通过 `from controller.app_context import get_theme_manager` 做隐式主题依赖。
- 16 个组件文件存在 `except Exception:` 的兜底捕获（共 22 处）。
- 10 个组件文件在组件内部直接 `setStyleSheet(...)`，与全局 QSS 方案并存。
- 包导出存在 API 不一致：`darkeye_ui.__all__` 含 `Input`，但模块里无该符号（`import darkeye_ui` 后不可用）。
- 自动化测试覆盖组件库很薄：目前仅发现 1 个手工测试文件引用组件库（`manual_tests/test_radarchartwidget.py`）。

## 2. 优先级建议

## P0（本周内，先做）

1. 修复公共 API 一致性
- 文件：`darkeye_ui/__init__.py`
- 建议：
  - 移除或替换 `__all__` 里的 `Input`。
  - 明确导出 `LineEdit / TextEdit / PlainTextEdit`（和 `components/__init__.py` 保持一致）。
  - 做一次 `__all__` 自检（CI 执行：`all(name in dir(module) for name in __all__)`）。

2. 收敛异常处理策略
- 现状：多个组件直接 `except Exception: pass`，错误被吞掉后很难排查主题失效、图标未加载等问题。
- 建议：
  - 仅捕获预期异常（如 `ImportError`、`RuntimeError`、`AttributeError`）。
  - 使用统一 logger 记录一次告警，不要静默吞掉。
  - 约定：UI 渲染路径可以降级，但必须可观测（日志/调试开关）。

3. 统一主题依赖注入方式
- 现状：大量组件在内部偷取全局 `controller.app_context`；测试和复用（独立项目）会变脆。
- 建议：
  - 以“显式注入 `theme_manager`”为主。
  - 保留兜底时，封装在一个 Mixin/Helper（不要每个组件重复 try-import）。
  - 为主题切换相关逻辑建立统一基类（含 `tokens()`、`on_theme_changed()` 模板）。

## P1（1~2 周）

1. 统一样式体系边界（QSS vs 组件内样式）
- 建议规则：
  - 基础控件（Button/Input/SpinBox/List/Table）走全局 QSS + `objectName`。
  - 强自绘控件（如雷达图、竖排文字）走 `paintEvent + tokens`。
  - 组件内 `setStyleSheet` 仅用于确实无法由全局 QSS 管理的局部弹层场景。
- 价值：避免主题切换时表现不一致，减少样式冲突。

2. Token 体系升级
- 文件：`darkeye_ui/design/tokens.py`, `darkeye_ui/styles/mymain.qss`
- 建议：
  - 将尺寸类 token 从字符串（如 `"12px"`）逐步拆成数值 token（如 `font_size_base_px: int`）。
  - 区分“语义 token”（primary/text/bg）与“组件 token”（button_height/table_header_bg）。
  - 增加 token 校验函数，防止主题扩展时漏字段。

3. 导入风格统一
- 现状：`radar_chart_widget.py` 使用绝对导入 `from darkeye_ui...`，其余多为相对导入。
- 建议：组件包内统一相对导入，减少路径/打包环境差异。

## P2（2~4 周）

1. 自动化测试补齐
- 建议新增：
  - 组件 smoke test：逐个实例化 + 主题切换。
  - 交互 test：Toggle、Completer、ColorPicker、Sidebar 等关键控件。
  - 自绘组件快照测试：`RadarChartWidget`、`CalendarHeatmap`、`VerticalTextLabel`。
- 工具建议：`pytest + pytest-qt`，并启用 `QT_QPA_PLATFORM=offscreen`。

2. 组件库发布工程化
- 建议：
  - 增加 `pyproject.toml`，明确组件库作为独立包的依赖与入口。
  - 建立语义化版本策略（至少内部版本号 + 变更记录）。
  - 约定 deprecation 周期（例如保留旧 API 1 个小版本）。

3. 文档补全
- 建议新增 `docs/components/`：每个组件最少包含
  - 用途
  - 关键参数
  - 主题行为
  - 最小示例
  - 已知限制

## 3. 两周落地计划（可直接排期）

第 1 周：稳定性与一致性
- Day 1-2：修复导出 API（`__all__`）、导入风格统一、异常处理最小改造。
- Day 3-4：抽出主题依赖 Mixin，替换重复的 `controller.app_context` 获取逻辑。
- Day 5：回归 demo，验证主题切换/图标/下拉弹层。

第 2 周：质量保障
- Day 1-2：补齐 smoke tests + 关键交互 tests。
- Day 3：补 2~3 个自绘控件快照测试。
- Day 4：整理组件文档模板并填充核心组件。
- Day 5：建立版本发布说明模板。

## 4. 验收标准（建议）

- `darkeye_ui.__all__` 全量可导入，无悬空符号。
- 主题切换时无静默失败，异常可通过日志定位。
- 组件测试覆盖核心控件，CI 可在 offscreen 环境稳定通过。
- 新组件接入时遵守统一规范：
  - 主题注入方式一致
  - 样式边界清晰
  - 文档与示例同步
