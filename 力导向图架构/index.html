二、架构问题与修改建议¶

2.1 核心问题：ForceViewOpenGL 是上帝对象（God Object）¶

现状：ForceViewOpenGL 拥有约 80+ 个成员变量，承担了以下所有职责：

  • 仿真线程管理与生命周期
  • OpenGL 初始化与绘制流水线
  • 鼠标/键盘输入处理
  • 图数据管理（节点、边、标签、颜色、半径）
  • 视图变换（平移/缩放）
  • 可见性裁剪与实例数据构建
  • MSDF 字体图集的异步构建与管理
  • 悬停/选择/拖拽状态机
  • 邻居 BFS 计算
  • 运行时图结构增删改

建议：逐步拆分为独立组件：

ForceViewOpenGL
├── GraphDataManager       — 图数据（节点/边/ID/标签/颜色/半径/邻居表）
│                            及 ID→索引映射 (HashMap)
├── SimulationController   — 仿真线程、PhysicsState、Simulation 的生命周期
├── ViewTransform          — pan/zoom 状态 + 坐标转换
├── InteractionHandler     — 鼠标事件 → 悬停/拖拽/选择状态机
├── VisibilityCuller       — 视口裁剪 + 分组(dim/base/hover)
├── EdgeBufferBuilder      — 边线/箭头顶点数据构建
├── NodeBufferBuilder      — 节点实例数据构建
└── TextSubsystem          — MsdfFontAtlas + MsdfTextRenderer + 布局缓存

优先级最高的拆分是 GraphDataManager（将 m_ids/m_labels/m_nodeColors/m_showRadiiBase/m_neighbors 和运行时增删逻辑抽出）和 SimulationController（将仿真线程和 mutex 封装）。

2.2 节点 ID 查找复杂度为 O(N)¶

现状：所有基于节点 ID (QString) 的操作都通过线性扫描 m_ids 来查找索引：

// ForceViewOpenGL_Runtime.cpp — 每次操作都是 O(N)
for (int i = 0; i < m_ids.size(); ++i) {
    if (m_ids[i] == nodeId) { ... }
}

在 getNodePosition、add_node_runtime、remove_node_runtime、add_edge_runtime、remove_edge_runtime 中均出现此模式。对于大图（10000+ 节点），这是严重的性能瓶颈。

建议：维护 QHash<QString, int> m_idToIndex，在 setGraph、运行时增删时同步更新。apply_diff_runtime 中已有局部的 buildIdToIndex() lambda，应将其提升为持久化成员。

2.3 边存在性检查复杂度为 O(E)¶

现状：PhysicsState::addEdge 需线性扫描所有边判断是否已存在：

for (size_t i = 0; i < edges.size(); i += 2) {
    if ((edges[i] == u && edges[i + 1] == v) || ...)
        return false;
}

建议：在 GraphDataManager 层维护 std::unordered_set<uint64_t> 的边集合（将 (u,v) 编码为 64 位键），O(1) 查重。

2.4 每帧重分配热路径内存¶

现状：updateEdgeLineBuffers、buildNodeInstanceData、buildTextVertices 在每帧调用 clear() + reserve() 重建顶点数据向量。虽然 std::vector::clear() 不释放内存，但 reserve 在容量不足时仍会重分配。

建议： - 使用 resize() + 手动管理写入偏移，避免反复 push_back - 或将这些缓冲区改为双缓冲（写完后 swap） - 对于 GPU 上传，考虑使用持久映射缓冲区 (persistent mapped buffer) 或环形缓冲

2.7 仿真线程信号发射的线程安全性¶

现状：simLoop() 在仿真线程中直接 emit Qt 信号：

emit tickTimeUpdated(elapsed);
emit alphaUpdated(alphaVal);
emit simulationStopped();

若信号-槽使用默认 AutoConnection，跨线程时会自动排队，但这依赖于 ForceViewOpenGL 有正确的线程亲和性。

建议： - 显式使用 Qt::QueuedConnection 连接这些信号 - 或改用线程安全的回调 + QMetaObject::invokeMethod(this, ..., Qt::QueuedConnection) 模式（如 setGraph 中已有的模式）

2.8 PhysicsState 职责过重¶

现状：PhysicsState 从一个纯数据结构膨胀为包含复杂运行时图操作的类（addNode、removeNode、addEdge、removeEdge），其中 removeNode 的 swap-last 逻辑约 60 行，且与上层（ForceViewOpenGL）的索引修复逻辑紧耦合。

建议： - PhysicsState 应保持为纯数据结构 + 简单的 init/accessor - 图结构修改操作提升到 GraphDataManager 层，统一处理物理数据和显示数据的同步

三、Bug 清单¶

Bug 7：PhysicsState::removeNode 的边自环过滤过于激进¶

位置：PhysicsState.h 第 206 行

if (u < 0 || v < 0 || u >= newN || v >= newN || u == v) continue;

问题：swap-last 后如果原来有一条边连接 index 和 last（假设 index != last），删除 index 节点时这条边应被移除（u == index 或 v == index 的检查已处理）。但如果原来有一条自环 (last, last)（虽然 addEdge 不允许），重映射后变成 (index, index)，会被 u == v 过滤掉。这不是实际 bug（因为 addEdge 阻止自环），但防御性检查 u == v 在 swap-last 语义下可能意外过滤掉合法边——需要确保这种情况不会发生。

现状：由于 addEdge 禁止自环，此处是安全的。但建议添加注释说明依赖关系。

四、性能优化建议¶

4.1 ManyBody Force 阈值调整¶

当前 N < 1000 使用 block-tiled O(N²)，N >= 1000 使用 Barnes-Hut。建议： - 在 500 ≤ N < 1000 范围增加 parallel block-tiled 路径（已声明 applyParallel 但未使用） - 考虑为 Barnes-Hut 实现 SIMD 加速

4.2 GPU Buffer 上传优化¶

当前每帧通过 glBufferData 重新分配并上传全部顶点数据。建议： - 使用 glBufferSubData 在缓冲区大小不变时仅更新数据 - 或使用 GL_MAP_PERSISTENT_BIT + triple-buffering 实现零拷贝

4.3 文本渲染按需更新¶

当前每帧都调用 buildTextVertices 重建文本顶点。如果视图和节点位置没有变化，可以缓存结果并跳过重建。添加脏标记 m_textVerticesDirty 在仿真 tick、视图变换、悬停变化时置位。
