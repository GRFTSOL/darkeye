> 基于开发白皮书 (develop.md) 与当前代码实现整理而成，用于指导产品规划与迭代优先级。

## 一、产品概述

### 1.1 产品定位

**DarkEye (暗眼)** 是一款 PC 端本地化、隐私导向的 AV 领域元数据管理与情报分析软件。用户可记录、收藏、分析个人观看偏好，通过力导向图探索作品与演员之间的关联，实现「个人撸得更爽」的目标。

**核心价值主张**：  
黄网有 UGC 打分、评论、热点推荐，但数据分散且隐私难控。爬虫类项目多，而**数据清洗、收藏管理、统计分析**的完整闭环工具稀缺。DarkEye 填补这一空白，提供本地优先、可控、可扩展的个人收藏与洞察系统。

### 1.2 目标用户

| 用户类型 | 典型需求 | 使用场景 |
|---------|---------|---------|
| 个人收藏者 | 整理番号、女优、标签 | 手动录入、爬虫抓取、批量管理 |
| 数据探索者 | 发现关联、偏好分析 | 力导向图、统计图表、智能推荐 |
| 隐私敏感用户 | 数据不外泄、快速伪装 | 本地存储、绿色模式、一键伪装 |
| 深度用户 | 记录行为、量化自我 | 撸管/做爱/晨勃记录、贤者报告 |

### 1.3 核心理念 (源自 develop.md)

| 理念 | 含义 |
|-----|------|
| **Curator (策展人)** | 不只收集番号，而是收藏与策展，挖掘有价值作品 |
| **Intelligence (情报)** | 从被动记录到主动洞察，挖掘偏好与关联（共演、出道轨迹等） |
| **Privacy First (绝对隐私)** | 本地优先架构，数据完全私有，支持伪装与加密 |
| **CJK Aesthetics (东方美学)** | 竖排 UI、东方审美，还原实体杂志沉浸感 |
| **Zero-Friction (零摩擦)** | 自动化采集与整理，最小化录入成本 |

### 2.2 功能详情与用户故事

#### 2.2.1 沉浸式采集 (Immersive Capture)

| 用户故事 | 验收标准 | 状态 |
|---------|---------|------|
| 作为用户，在 JavLibrary/JavDB 浏览时，希望插件图标显示本地是否已收藏 | 插件调用 `/api/v1/check_existence`，图标状态实时更新 | ✅ |
| 作为用户，希望点击插件即可将番号推送到 DarkEye | 插件 POST 到 `/api/v1/capture`，主窗口弹出 AddQuickWork 确认 | ✅ |
| 作为用户，希望采集的数据自动爬取元数据后进入待确认流程 | 爬虫多源抓取，合并结果后写入或展示确认 | ✅ |
| 作为用户，希望有一个「收件箱」待处理队列 | 待明确设计 | ⏳ Phase 1 |

#### 2.2.2 智能化管理 (Intelligent Management) 

| 用户故事 | 验收标准 | 状态 |
|---------|---------|------|
| 作为用户，希望用卡片流浏览作品/女优 | 瀑布式布局、懒惰加载、封面卡片 | ✅ |
| 作为用户，希望通过力导向图探索女优、作品、导演关系 | ForceView 力导向图，支持筛选与交互 | ✅ |
| 作为用户，希望多维度筛选作品 | 番号、女优、导演、男优、标签、标题、剧情、片商 | ✅ |
| 作为用户，希望标签支持复杂逻辑（包含/排除） | TagSelector 支持多选与逻辑 | ✅ |
| 作为用户，希望首页有拟物化书架/无限卡片流 | CoverBrowser 封面流，可随机/智能推荐 | ✅ |
| 作为用户，希望卡片详情和标题支持竖排 | 部分控件支持竖排 | ⏳ Phase 1 |

#### 2.2.3 量化自我 (Quantified Self)

| 用户故事 | 验收标准 | 状态 |
|---------|---------|------|
| 作为用户，希望记录撸管、做爱、晨勃行为 | 对应对话框录入，私有库存储 | ✅ |
| 作为用户，希望看到最喜欢的女优统计（30/180/365 天） | PersonalDataPage、SwitchHeapMap | ✅ |
| 作为用户，希望看到收藏中未观看的去化周期 | 统计图表 | ✅ |
| 作为用户，希望收到月度偏好报告 | 贤者报告模块 | ❌ Phase 3 |

#### 2.2.4 极致安全 (Ultimate Privacy)

| 用户故事 | 验收标准 | 状态 |
|---------|---------|------|
| 作为用户，希望所有数据存储在本地 | 数据在 `resources/`，可配置路径 | ✅ |
| 作为用户，希望图片可打码展示 | 绿色模式开关，状态栏切换 | ✅ |
| 作为用户，希望一键伪装界面 | Excel/代码编辑器伪装 | ❌ Phase 3 |
| 作为用户，希望私有库可加密 | SQLCipher 或类似方案 | ❌ Phase 3 |

## 三、非功能需求

### 3.1 性能

| 需求 | 指标 |
|-----|------|
| 启动时间 | 含 Sim 进程、图初始化、DB 迁移后，< 2s 进入主界面 |
| 卡片加载 | 懒惰加载，滚动时按需渲染，避免一次性加载全部 |
| 力导向图 | 大图场景下保持可交互帧率 |
| 数据库 | SQLite WAL 模式，支持并发读 |

### 3.2 兼容性

| 维度 | 要求 |
|-----|------|
| 操作系统 | Windows（主要），兼顾 macOS / Linux |
| Python | 3.12 |
| 打包 | PyInstaller / Nuitka 绿色可移动包 |

### 3.3 可维护性

| 维度 | 要求 |
|-----|------|
| 代码风格 | Type Hints、模块化、Repository 逐步迁移 |
| 测试 | 核心逻辑有单元测试 |
| 文档 | MkDocs 构建，API 与架构文档同步更新 |

### 3.4 可扩展性

| 维度 | 要求 |
|-----|------|
| 爬虫 | 模块化，支持多源（JavLib、JavTxt、Fanza、AV 蛋蛋维基等） |
| 插件 | 未来支持第三方爬虫与数据分析插件 |
| 配置 | settings.ini 管理路径、快捷键等 |

## 四、技术约束与实现现状

### 4.1 技术栈

| 层级 | 技术选型 | 说明 |
|-----|---------|------|
| UI | PySide6 | Qt for Python，QGraphicsView 实现力导向图与卡片流 |
| 本地服务 | FastAPI | 处理插件请求、SSE 推送 |
| 数据库 | SQLite | public.db（作品等）+ private.db（撸管/做爱/晨勃） |
| 爬虫 | 自研 + 多源 | JavLib、JavTxt、AV 蛋蛋维基、Fanza（部分） |
| 图计算 | NetworkX | 力导向图节点与边生成 |

### 4.2 数据模型概要

- **公共库**：作品 (work)、女优 (actress)、男优 (actor)、标签 (tag)、制作商 (maker)、关联表等  
- **私有库**：撸管 (masturbation)、做爱 (lovemaking)、晨勃 (sexual_arousal)、喜欢作品/女优等  
- **配置**：settings.ini 管理路径、窗口状态、快捷键等  

### 4.3 已知限制

- 黄网在挖掘新片上存在劣势，无 UGC 打分与热点推荐  
- 爬虫依赖第三方站点稳定性，部分信息可能抓取不到  
- 日文标题与剧情需依赖翻译服务（如机翻）辅助理解

## 五、产品路线图

### Phase 1: 策展人更新 (The Curator Update) — 当前重点

| 需求 | 描述 | 优先级 |
|-----|------|--------|
| 无限卡片流首页 | 替换现有欢迎页，高性能水平滚动 | P0 |
| CJK 竖排 UI | 卡片详情、标题竖排排版 | P1 |
| Inbox 模式重构 | 待处理队列，改造手动录入流程 | P0 |

### Phase 2: 情报网更新 (The Intelligence Update)

| 需求 | 描述 | 优先级 |
|-----|------|--------|
| 插件双向通信 | 网页端实时显示本地收藏状态 | P0 |
| 关注订阅 | 女优/片商新作预告推送 | P1 |
| 多源数据融合 | ID 解析、去重、元数据准确性提升 | P1 |

### Phase 3: 隐形人更新 (The Invisible Update)

| 需求 | 描述 | 优先级 |
|-----|------|--------|
| Excel 伪装模式 | 一键 UI 切换 | P0 |
| 数据加密 | 私有库加密支持 | P1 |
| 贤者报告 | 行为分析与可视化 | P1 |

### 长期目标

- **i18n**：简中、繁中、日文、韩文  
- **社区生态**：插件市场、爬虫规则与模型分享  
- **2.0 UGC / 分布式**：共识修正、P2P 同步（见 future.md）  
- **3.0 机器学习推荐**：基于行为与内容的推荐算法

## 六、成功指标

| 指标 | 说明 | 目标 |
|-----|------|------|
| 录入效率 | 从浏览器到入库的步骤数 | 插件一键 + 确认 ≤ 3 步 |
