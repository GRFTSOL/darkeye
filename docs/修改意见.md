# DarkEye 项目架构评估与修改意见（在“可直接读写数据库”前提下）

## 1. 结论摘要
- 当前项目属于“桌面单体应用（PySide6）+ 内嵌 FastAPI + SQLite（公库/私库）+ 浏览器扩展 + C++ OpenGL 图引擎”的混合架构。
- 方向可行，功能覆盖面广，但可维护性风险主要集中在三处：数据库初始化/迁移链路、数据访问方式不统一、跨线程/跨进程链路的契约不清晰。
- 你已接受“直接读写数据库”，因此不建议强行上 ORM 或重服务化；建议走“保留直连 SQL + 统一连接契约 + 强迁移治理 + 关键链路补测试”的路线。

## 2. 本次评估范围
- 文件扫描统计（项目内）：
  - `*.py`: 271
  - `*.cpp/*.h/*.hpp`: 31
  - `*.js`: 9
  - `*.sql`: 32
- 重点阅读模块：
  - 入口/启动：`main.py`, `config.py`, `app_context.py`
  - UI/路由：`ui/main_window.py`, `ui/navigation/router.py`, `ui/pages/*`
  - 服务端：`server/app.py`, `server/launcher.py`, `server/bridge.py`
  - 爬虫链路：`core/crawler/*`
  - 数据库：`core/database/*`, `core/database/query/*`, `resources/sql/*`
  - 图模块：`core/graph/*`, `ui/pages/ForceDirectPage.py`
  - C++绑定：`cpp_bindings/forced_direct_view/*`, `cpp_bindings/color_wheel/*`
  - 扩展：`extensions/firefox_capture/*`

## 3. 架构现状评价

### 3.1 优点
- 模块划分总体清晰：UI、数据库、爬虫、图渲染、扩展各自有目录边界。
- C++ `forced_direct_view` 组件分层较好（Simulation/Forces/Renderer/Text/Runtime 分文件），工程化程度高。
- 浏览器扩展与本地 FastAPI 的通信链路完整（SSE + HTTP），端到端闭环可用。
- SQLite 查询层（`core/database/query/*`）内容丰富，统计能力较强。

### 3.2 主要问题

#### P0（优先最高，影响稳定性）
- 数据库初始化与迁移链路存在“不可依赖”点：
  - `core/database/init.py` 中 `init_public_db` 仍为 `pass`。
  - `resources/sql/initTABLE.sql` 中存在明显结构问题（例如外键语法缺分隔、`prefix_maker_relation` 引用了 `maker` 但前文为 `manufacturer` 命名）。
  - `core/database/migrations.py` 升级逻辑偏样例化，缺少可追溯、可回放的真实迁移序列。
- 数据访问路径不统一：
  - 同时存在 `get_connection(...)` 与多处 `sqlite3.connect(...)`（尤其 UI 页面层），连接策略（只读、`foreign_keys`、`busy_timeout`）不一致。
- 服务/爬虫关键分支未收敛：
  - `server/app.py` 与 `core/crawler/CrawlerManager.py` 中仍有 `pass` 分支和 `print(...)` 路径，`javdb/fanza` 结果处理不完整。

#### P1（中优先，影响维护成本）
- UI 层直接承担了较多业务编排和数据访问，页面职责偏重，导致变更扩散。
- 系统依赖多个全局/单例对象（Router、GraphManager、CrawlerManager、bridge、signal bus 等），降低可测试性。
- 图模块跨线程复杂（Qt 主线程 + 渲染/仿真线程 + Python diff 推送），当前有锁但缺少明确“接口契约文档 + 回归用例”。
- Firefox 扩展对站点 DOM 结构耦合高，抗网站改版能力弱。

#### P2（次优先，影响长期演进）
- 自动化测试覆盖不足：`tests` 仅少量用例，缺数据库迁移/服务接口/图运行时回归。
- `junk/` 与 `manual_tests/` 体量较大，实验代码与主线边界不够清晰。

## 4. 在“允许直接读写数据库”前提下的改造原则
- 不引入重型 ORM，不强制 Repository/Service 全量重构。
- 保留 SQL 直写能力，但必须统一连接入口和事务策略。
- 允许 UI 直接查询数据库，但要满足三条硬约束：
  - 统一连接工厂（只读/读写、PRAGMA、timeout 一致）。
  - 写操作走明确写接口（`insert/update/delete` 或约定写函数），避免页面内散落拼接写 SQL。
  - 迁移脚本版本化、可回放、可验证。

## 5. 修改建议（落地版）

### 5.1 P0 先做（1-2 周）
1. 修复 DB 启动基线
- 完成 `init_public_db`，并把公库初始化纳入统一启动流程。
- 校验并修复 `resources/sql/initTABLE.sql` 的结构错误（外键/表名一致性/依赖顺序）。

2. 建立可执行迁移体系
- 建议新增：
  - `resources/sql/migrations/public/0001_*.sql`
  - `resources/sql/migrations/private/0001_*.sql`
- `core/database/migrations.py` 改为按版本顺序执行并记录（失败回滚、幂等检查、日志可追踪）。

3. 统一连接策略
- 约定：业务代码禁止直接 `sqlite3.connect(...)`（迁移/备份脚本可例外）。
- 全量改为 `get_connection(...)` 或统一封装（只读与读写明确分离）。

4. 收敛未完成分支
- `server/app.py`、`core/crawler/CrawlerManager.py` 的 `pass` 分支要么补全要么显式返回“不支持”。
- 将 `print(...)` 全量替换为标准日志，保留结构化字段（serial/source/web/error）。

### 5.2 P1 跟进（2-4 周）
1. 页面减负但不“重服务化”
- 保留页面直查 DB；把高复用查询下沉到 `core/database/query/*`，避免同类 SQL 在页面重复。
- 为“复杂写入流程”（如爬虫入库、多表联动）建立薄流程函数（不是重 Service 层）。

2. 图模块契约化
- 固化 `graph_diff` 数据结构文档（字段、op 类型、兼容策略）。
- 为 `apply_diff_runtime` 建立 Python 侧回归测试（增删节点/边、异常输入、并发触发）。

3. 扩展层稳健性
- 为 SSE 断连重连、DOM 查询失败、站点改版增加分级错误码与回退策略。
- 把站点脚本抽象出统一适配层（最小限度，不做过度设计）。

### 5.3 P2 持续改进（并行）
1. 测试基线
- 至少补齐：
  - 数据库初始化 + 迁移可回放测试
  - Server 核心接口契约测试（`/health`, `/check_existence`, `/crawler-result`）
  - 爬虫合并逻辑单测（多源冲突合并）

2. 目录治理
- 明确 `junk/`、`manual_tests/` 的生命周期：保留、迁移或归档，并在 README/开发文档中声明。

## 6. 建议验收标准（可量化）
- 启动阶段数据库任务可在“空库 + 老库”两种场景稳定通过。
- 业务层 `sqlite3.connect(...)` 数量显著收敛（迁移/备份脚本除外）。
- 迁移脚本可从 `0001` 顺序回放到当前版本，失败时可定位具体脚本。
- `server/crawler` 不再存在未处理 `pass` 主分支，关键错误均有日志。
- 自动化测试覆盖 DB 初始化/迁移、Server 契约、核心合并逻辑。

## 7. 最终建议
- 这套项目不需要“推倒重来”，也不需要为了架构“纯洁性”牺牲迭代速度。
- 直接读写数据库的做法可以继续，但要尽快补齐“三件套”：
  - 统一连接
  - 可靠迁移
  - 关键链路测试
- 只要先完成 P0，本项目可维护性会明显上一个台阶。
