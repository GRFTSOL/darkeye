# Python 代码整体修改意见

## 扫描范围与结论
- 已扫描并读取全部 `*.py` 文件，共 `308` 个文件、`58,397` 行。
- 业务代码（排除 `manual_tests/`、`tests/`、`exp/`、`cpp_bindings/.../3rdparty/`）为 `261` 个文件、`46,201` 行。
- 第三方 vendored 代码中有 `3` 个 Python2 脚本在 Python3 下语法不兼容（`freetype/src/tools`），应从本项目质量门禁中排除。

## 主要问题（按影响排序）

### P0：稳定性与可维护性风险
- 异常处理过宽：业务代码中有 `179` 处 `except Exception`，另有 `1` 处裸 `except`（`controller/ShortcutRegistry.py:53`）。
- 异常吞掉问题：`except ...: pass` 约 `8` 处，排障信息会丢失（如 `controller/MessageService.py:65`、`ui/pages/ManagementPage.py:73`）。
- SQL 语句拼接：检测到 `execute(f"...")` 形式（如 `core/database/backup_utils.py:78`、`ui/base/SqliteEditableTableModel.py:45`），表名/库名若来自外部输入会有注入风险。
- 包结构命名问题：存在 `ui/widgets/selectors/__init.py`（缺少一个下划线）和 `scripts/pack..py`（双点文件名），易引发导入和工具链兼容问题。

### P1：规模膨胀导致的重构压力
- 超大文件较多（业务代码中 `>=800` 行文件 7 个）：
  - `ui/pages/management/AddWorkTabPage3.py`（1482 行）
  - `core/dvd/DvdShelfView.py`（1314 行）
  - `ui/widgets/image/FanartStripWidget.py`（1255 行）
  - `ui/widgets/selectors/TagSelector5.py`（1046 行）
  - `core/database/update.py`（956 行）
- 长函数偏多：业务代码中 `>=120` 行函数约 `17` 个，UI 初始化与数据装配逻辑耦合明显。

### P1：可观测性与规范一致性
- 业务代码 `print(...)` 约 `175` 处，日志渠道分散，线上定位困难。
- `core/database/__init__.py:1` 使用 `from .query import *`，会增加命名污染和静态分析难度。

### P2：工程化基础能力不足
- 类型标注不足：业务代码函数约 `2340` 个，其中未完整标注签名约 `1363` 个。
- 文件命名不统一：非 snake_case 文件约 `97` 个（以 UI 层最集中），对跨平台和自动化工具不友好。
- 测试占比偏低：测试代码约 `4,271` 行，和业务代码行数比约 `9.2%`，且 `manual_tests/` 比重较高。

## 建议的落地顺序
1. 先做“异常治理 + SQL 安全 + 命名修复”（P0），优先覆盖 `core/database/*`、`core/crawler/*`、`server/app.py`。
2. 建立统一错误策略：禁止裸 `except`、限制 `except Exception`、禁止静默 `pass`（除明确注释说明的场景）。
3. 建立数据库访问约束：值参数必须参数化；标识符（表名/字段名）仅允许白名单映射。
4. 对超大模块按“UI 构建 / 状态管理 / 数据访问”拆分，先拆 `AddWorkTabPage3.py` 与 `DvdShelfView.py`。
5. 将业务层 `print` 迁移到统一 logger（保留脚本目录 `scripts/` 中的 CLI 输出）。
6. 逐步补齐类型与测试：先从数据库接口、服务层和跨模块公共函数开始。

## 可量化验收目标（建议）
- `except Exception`：`179 -> <50`，裸 `except` 清零。
- 业务代码 `print`：`175 -> <20`（仅保留明确调试开关）。
- `>=800` 行文件：`7 -> <=2`。
- 关键路径（数据库/爬虫/页面加载）补充可重复自动化测试。
